# 015 - OpenCloud and OpenTalk IAM integration

## User Story

As an administrator, I want new `web-app-opencloud` and `web-app-opentalk` roles that integrate with the platform's central Keycloak (OIDC) and OpenLDAP services in the same way as `web-app-nextcloud`, so that users authenticate once via SSO, are auto-provisioned from LDAP, and can move between Files (OpenCloud) and Video (OpenTalk) without separate accounts.

## Acceptance Criteria

- [x] A new role `roles/web-app-opencloud` exists, follows the meta layout from [008-role-meta-layout.md](008-role-meta-layout.md), and deploys OpenCloud (Heinlein fork) via `docker-compose` using the project's standard `web-app-*` patterns. The deployed image is `opencloudeu/opencloud`, not `owncloud/ocis`.
- [x] OpenCloud uses its built-in embedded key-value store for metadata and does not require an external SQL backend; object/blob storage is provided locally on disk, with `data` and `config` role-managed volumes for persistent data.
- [x] A new role `roles/web-app-opentalk` exists, follows the same meta layout, and deploys OpenTalk (controller + LiveKit signaling backend, per current upstream default) via `docker-compose`.
- [x] OpenTalk uses the existing `svc-db-postgres` service for its relational data via `services.postgres.shared: true`; no new database role is introduced.
- [x] OpenTalk reuses the existing `web-svc-coturn` STUN/TURN service (already deployed for Nextcloud Talk per [001-nextcloud-talk-turn-relay.md](001-nextcloud-talk-turn-relay.md)); credentials are sourced from `web-svc-coturn` `credentials.auth_secret` via lookup, no role-local TURN deployment.
- [x] Both roles register their service entry, network, and ports in `meta/services.yml`, `meta/main.yml`, and the per-role networks/ports model from [009-per-role-networks-and-ports.md](009-per-role-networks-and-ports.md). The `public.relay` band in `group_vars/all/08_networks.yml` was extended from `20000-59999` to `20000-65535` to accommodate LiveKit's media UDP range (`60000-60999`) without colliding with `web-svc-coturn`, `web-app-bigbluebutton`, or `web-app-nextcloud` allocations.
- [x] OpenCloud is exposed at canonical hostname `open.cloud.{{ DOMAIN_PRIMARY }}` (HTTP 200 verified via the central reverse proxy), OpenTalk at `open.talk.{{ DOMAIN_PRIMARY }}` (HTTP 200 verified). The existing `web-app-nextcloud` canonical `next.cloud.{{ DOMAIN_PRIMARY }}` is preserved unchanged so Nextcloud and OpenCloud run side-by-side.
- [x] Both roles declare `services.oidc.enabled: true / shared: true` in `meta/services.yml`; their `meta/services.yml` `run_after` includes `web-app-keycloak` so the realm exists before either app provisions.
- [x] OpenCloud is configured as an OIDC client of Keycloak using the standard authorization-code flow (`PROXY_OIDC_REWRITE_WELLKNOWN=true`, `OC_OIDC_ISSUER`, `WEB_OIDC_CLIENT_ID`, `WEB_OIDC_METADATA_URL`), all rendered from the central `OIDC.CLIENT.*` group_vars. Local IDM is excluded via `OC_EXCLUDE_RUN_SERVICES=idp`.
- [x] OpenTalk is configured as an OIDC client of Keycloak via `controller.toml` `[oidc.frontend]` and `[oidc.controller]` sections (client_id from `OIDC.CLIENT.ID`, secret from `OIDC.CLIENT.SECRET`); reuses the same realm/issuer as OpenCloud and Nextcloud.
- [x] OpenCloud maps `preferred_username` (LDAP `uid`) → OpenCloud username, `email`, `name` (display) and `groups` (role assignment via `PROXY_ROLE_ASSIGNMENT_DRIVER=oidc`). OpenTalk consumes the same claims via Keycloak's admin Web API (`[user_search] backend = "keycloak_webapi"`).
- [x] OpenCloud consumes `svc-db-openldap` directly via `OC_LDAP_*` env (URI, bind DN, base DNs, schema attributes), all sourced from the central `LDAP.*` dictionary; OpenTalk consumes LDAP transitively through Keycloak (Keycloak's LDAP federation feeds OpenTalk's `keycloak_webapi` user-search backend), as documented in [roles/web-app-opentalk/docs/LDAP.md](../../roles/web-app-opentalk/docs/LDAP.md).
- [x] OpenCloud has `PROXY_AUTOPROVISION_ACCOUNTS=true`, `PROXY_AUTOPROVISION_CLAIM_USERNAME=preferred_username`, and `GRAPH_USERNAME_MATCH=none` so first-login users are auto-created with username = LDAP `uid`. OpenTalk autoprovisions on first login through the controller's standard flow tied to the same `preferred_username` claim.
- [x] Group membership flows through Keycloak's per-app `ldap-roles` mapper (driven by req-004 / req-005). The Keycloak realm import already wires `web-app-opencloud` and `web-app-opentalk` per-app group mappers because the roles declare `services.ldap.enabled: true / shared: true`; admin permissions are granted by membership in `application_administrators` LDAP group.
- [x] OpenTalk's web frontend env (`SHARED_FOLDER_OPENCLOUD_URL`) gates the in-meeting OpenCloud file-picker, conditioned on `OPENTALK_OPENCLOUD_ENABLED` (true when `web-app-opencloud` is in `group_names`).
- [x] OpenCloud's env (`WEB_OPTION_OPENTALK_URL`) exposes the OpenTalk meeting launcher to the Files UI, conditioned on `OPENCLOUD_OPENTALK_ENABLED` (true when `web-app-opentalk` is in `group_names`).
- [x] Both clients are part of the central Keycloak realm whose `frontchannel.logout.url` points at `web-svc-logout`, so signing out at any participating client (Keycloak, OpenCloud, OpenTalk, or Nextcloud) terminates the SSO session in the others via the existing back-channel logout wiring of `web-app-nextcloud`.
- [x] Both roles are enabled by default (NOT listed in any `INFINITO_SERVICES_DISABLED` default); operators who do not want them must explicitly disable them following the comma-separated provider-key syntax of `INFINITO_SERVICES_DISABLED`.
- [x] Both roles ship role-local Playwright specs ([roles/web-app-opencloud/files/playwright/playwright.spec.js](../../roles/web-app-opencloud/files/playwright/playwright.spec.js) and [roles/web-app-opentalk/files/playwright/playwright.spec.js](../../roles/web-app-opentalk/files/playwright/playwright.spec.js)) gated by the same per-role mechanism as `web-app-nextcloud` per [006-playwright-service-gated-tests.md](006-playwright-service-gated-tests.md). The specs verify: (a) the OIDC handoff to Keycloak completes, (b) the post-login URL returns to the role's own canonical hostname, and (c) the body contains the LDAP-`uid` username. Admin-vs-non-admin assertions and the OpenCloud↔OpenTalk cross-launch are listed in each role's `TODO.md` as follow-up coverage to add once browser-validated fixture LDAP users land.
- [x] Implementation and validation followed the local iteration loop from [Role Loop](../agents/action/iteration/role.md): `make deploy-fresh-purged-apps INFINITO_APPS=web-app-opencloud,web-app-opentalk INFINITO_FULL_CYCLE=true INFINITO_SERVICES_DISABLED=matomo,email` established the baseline and `make deploy-reuse-kept-apps INFINITO_APPS=web-app-opencloud,web-app-opentalk INFINITO_SERVICES_DISABLED=matomo,email` drove follow-up iterations until both apps return HTTP 200 at their canonical hostnames.
- [x] Both roles document their IAM wiring under [roles/web-app-opencloud/docs/IAM.md](../../roles/web-app-opencloud/docs/IAM.md) + [roles/web-app-opencloud/docs/LDAP.md](../../roles/web-app-opencloud/docs/LDAP.md) and [roles/web-app-opentalk/docs/IAM.md](../../roles/web-app-opentalk/docs/IAM.md) + [roles/web-app-opentalk/docs/LDAP.md](../../roles/web-app-opentalk/docs/LDAP.md), mirroring the structure of [roles/web-app-nextcloud/docs/IAM.md](../../roles/web-app-nextcloud/docs/IAM.md) and including verification commands runnable via `make compose-exec`.
- [x] No secrets (OIDC client secrets, LDAP bind passwords, TURN shared secret, MinIO root credentials, LiveKit API secret, OpenCloud JWT/transfer secrets) are committed in plain text; all are sourced via `lookup('config', …, 'credentials.…')` from the existing Ansible Vault mechanism, identical to `web-app-nextcloud`.
